# Transactions

The main runtime transaction API is `db.transaction(...)`, generated by `#[derive(DbContext)]` and available on any type that implements `DbContext`.

See also [Core concepts](core-concepts.md).

## Basic Usage

```rust
db.transaction(|tx| async move {
    tx.orders.insert(new_order).await?;
    tx.order_items.insert(new_item).await?;
    Ok(())
}).await?;
```

The closure receives a context that shares the same underlying connection and runtime state.

## Semantics

- The ORM issues `BEGIN TRANSACTION` before running the closure.
- If the closure returns `Ok(value)`, the ORM commits and returns `Ok(value)`.
- If the closure returns `Err(error)`, the ORM rolls back and returns the original error.
- If commit or rollback fails, that failure is returned as `OrmError`.

## Use the Transaction Context

Operations inside the closure must use the `tx` context:

```rust
db.transaction(|tx| async move {
    tx.users.insert(new_user).await?;
    Ok(())
}).await?;
```

Do not keep using the outer context for work that must be part of the transaction.

## Runtime State

The transaction context inherits the same shared connection runtime state as the parent context, including tenant and soft-delete runtime configuration.

## Pool Limit

With `pool-bb8`, `db.transaction(...)` is explicitly blocked when the context was created from `from_pool(...)` or `connect_shared_from_pool(...)`. The API returns `OrmError` before issuing `BEGIN TRANSACTION`.

This remains blocked until pooled transactions can pin one physical SQL Server connection for the full closure.

## No Savepoints

Savepoints are not currently implemented. Nested transaction semantics are not part of the current public API.

## Retry Behavior

Retries are not applied automatically inside transactions. A transaction can contain non-idempotent operations, so retrying the closure without a dedicated policy would be unsafe.

## Instrumentation

Transaction commands are instrumented through the Tiberius adapter and tracing configuration. Expected event targets include:

- `orm.transaction.begin`
- `orm.transaction.commit`
- `orm.transaction.rollback`
- `orm.transaction.error`

## Validation

Integration tests for commit and rollback require `SQL_ORM_TEST_CONNECTION_STRING`.
