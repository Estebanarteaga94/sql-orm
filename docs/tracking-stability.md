# Tracking Stability Criteria

This document defines the minimum criteria required before `Tracked<T>`,
`EntityState`, `find_tracked(...)`, `add_tracked(...)`, `remove_tracked(...)`
and `save_changes()` can stop being documented as experimental.

It does not stabilize the current implementation. It is the acceptance
contract for the remaining Etapa 21 work.

The concrete unit-of-work design for the next implementation slices is defined
in [`tracking-unit-of-work.md`](tracking-unit-of-work.md).

## Current Experimental Baseline

The current implementation is useful but intentionally narrow:

- `Tracked<T>` stores an original snapshot, current value and `EntityState`.
- `find_tracked(...)` and `add_tracked(...)` register wrappers in a context
  registry.
- `save_changes()` is generated by `#[derive(DbContext)]` and calls each
  `DbSet` in three phases: `Added`, `Modified`, then `Deleted`.
- Pending `Added`, `Modified` and `Deleted` entries remain registered through
  registry-owned snapshots after their wrappers are dropped or consumed.
- Detached loaded identities can reattach to registry-owned original/current
  snapshots, while a second live handle for the same persisted identity is
  rejected.
- Registry-owned snapshots are updated when no-op changes are accepted or
  persisted rows are synchronized, even after the original wrapper has been
  dropped.
- Persisted identity collisions are rejected both against live entries and
  detached registry-owned entries.
- Mutable access marks `Unchanged` values as `Modified` immediately.
- `remove_tracked(...)` on an `Added` wrapper cancels the pending insert.
- Persistence reuses the existing `DbSet` insert/update/delete paths.
- Public tracking persistence is still centered on simple primary keys.
- Navigation includes and explicit loads do not register related entities
  automatically and do not persist relationship changes.

Those facts are the current stabilization baseline, not final release
guarantees. The public label should remain experimental until the final Stage
21 validation, release documentation and compatibility decision are complete.

## Stability Bar

Tracking can be called stable only when the API has deterministic behavior for:

- ownership of tracked entities,
- duplicate tracking of the same entity,
- state transitions,
- operation ordering,
- transaction boundaries,
- policy integration,
- optimistic concurrency,
- errors,
- primary key scope,
- navigation interaction,
- and public documentation/tests.

Renaming rustdoc from experimental to stable is the final step, not the first
one.

## API Guarantees

A stable API must guarantee:

- `Tracked<T>` exposes only intentional public state transitions.
- Users can inspect the current state without relying on hidden registry
  details.
- Users can explicitly mark an entity as `Modified`, `Deleted` or `Unchanged`.
- Users can detach one entity from the context tracker.
- Users can clear the context tracker.
- Dropping a `Tracked<T>` wrapper does not silently discard pending context
  changes unless that is the documented stable behavior and covered by tests.
- Cloning or moving a `Tracked<T>` value has documented identity and registry
  semantics.
- `save_changes()` returns the number of persisted operations with a precise
  definition of what counts as one operation.
- `save_changes()` is idempotent for `Unchanged` entities and for already
  synchronized tracked entries.

The preferred direction for Etapa 21 is a context-owned registry. `Tracked<T>`
may remain a wrapper, but the unit of work must not depend on wrapper lifetime.

## Semver Guarantees

After stabilization:

- Public method names and signatures for stable tracking APIs are semver
  protected.
- `EntityState` variant names are semver protected unless explicitly marked
  `#[non_exhaustive]` before stabilization.
- Error categories observable through `OrmError` are semver relevant.
- Runtime guarantees about duplicate tracking, detach, operation ordering and
  transaction behavior are part of the public contract.
- Internal registry shape, SQL AST internals and generated macro helper names
  remain private implementation details.

If an intended behavior cannot be semver-protected, it must stay documented as
a limit instead of being implied as stable.

## Identity And Duplicate Tracking

The stable tracker must define an identity key for each tracked row.

Minimum accepted key:

- entity Rust type,
- schema,
- table,
- primary key values in declared metadata order.

Stable behavior must avoid silent duplicate in-memory entries for the same
persisted row. The first stable-cut policy is:

- one live `Tracked<T>` handle is allowed per persisted identity per context;
- if the existing registry entry still exists but its live wrapper was dropped
  or consumed, `find_tracked(...)` reattaches a new wrapper to the
  registry-owned snapshots;
- if another live wrapper is still attached, `find_tracked(...)` rejects the
  duplicate with a clear `OrmError` telling the caller to detach or drop the
  existing handle before loading the same identity again;
- navigation materialization may reuse registry-owned snapshots for related
  entities, but it does not create additional live tracked handles.

## State Transitions

Stable state transitions must be explicit and tested:

- `Unchanged -> Modified`
- `Unchanged -> Deleted`
- `Modified -> Unchanged`
- `Modified -> Deleted`
- `Added -> Unchanged` after successful insert
- `Added -> Deleted` as local cancellation, with no database delete
- `Deleted -> detached` after successful delete
- `Deleted -> Unchanged` only if an explicit restore API exists

No-op saves must not issue unnecessary SQL for entries that are effectively
unchanged.

## Change Detection

Stable tracking must not treat every mutable borrow as a guaranteed database
update unless that behavior is deliberately kept and documented as the stable
tradeoff.

Preferred requirement:

- compare persisted snapshots against current values structurally enough to
  skip updates when no persisted column changed,
- avoid runtime reflection,
- ignore navigation wrapper state unless relationship persistence is explicitly
  in scope,
- ignore computed, identity, rowversion and non-updatable columns for update
  detection.

If structural comparison requires generated code, it belongs in
`sql-orm-macros` and public traits exposed by `sql-orm`, not in `query`,
`sqlserver` or `tiberius`.

## Operation Ordering

`save_changes()` must order operations deterministically.

Minimum stable rule for simple foreign keys:

- insert principals before dependents when both are `Added`,
- update existing rows after inserts when FK dependencies require generated
  keys,
- delete dependents before principals when both are `Deleted`,
- keep a deterministic fallback order by context field order and entity
  metadata when no dependency exists.

If relationship graph ordering is not implemented in the first stable cut, the
stable contract must reject ambiguous graphs early with a clear error instead
of relying on SQL Server constraint failures.

## Transactions

Stable `save_changes()` must define its transaction behavior.

Acceptable stable options:

- always open an internal transaction when no transaction is active,
- reuse the active `db.transaction(...)` connection when called inside a
  transaction,
- or explicitly document that the caller must wrap `save_changes()` in
  `db.transaction(...)` for atomicity.

The final contract must be backed by tests for:

- successful commit of multiple operations,
- rollback after a mid-save error,
- interaction with direct connections,
- and the documented behavior for pooled contexts while Etapa 22 remains
  blocked.

Partial persistence without documentation and tests is not acceptable.

## Concurrency And Errors

Stable tracking must preserve existing optimistic concurrency behavior:

- `rowversion` mismatches surface as `OrmError::ConcurrencyConflict`,
- a missing row during update/delete produces a documented error category,
- tenant mismatch or missing tenant fails before SQL compilation,
- duplicate tracked identity uses a documented error or merge behavior,
- unsupported primary key shapes fail with a stable, actionable error,
- invalid state transitions fail before database execution.

Errors must not require matching long free-form strings in user code. If the
current `OrmError` shape is too generic for stable tracking, Etapa 21 must add
typed error categories before removing the experimental label.

## Policy Integration

Stable `save_changes()` must reuse the same policy paths as explicit CRUD:

- `tenant` insert auto-fill and mismatch rejection for `Added`,
- tenant predicates for `Modified` and `Deleted`,
- `soft_delete` update semantics for tracked deletes,
- `SoftDeleteProvider`, `SoftDeleteValues` and request values for deletes,
- `AuditProvider`, `AuditValues` and request values for insert/update,
- `rowversion` predicates and refreshed tokens where SQL Server returns them.

Tracking must not create a second persistence pipeline that bypasses policy
runtime behavior.

## Primary Key Scope

Before stabilization, the project must choose one of two contracts:

- support composite primary keys in tracking and cover them with tests,
- or keep tracking limited to simple primary keys with a stable, explicit
  runtime error and documentation that no route silently misbehaves.

The first stable cut may choose the second option, but it must be framed as a
supported limit, not as experimental uncertainty.

## Active Record Interaction

Tracking must remain layered on top of `DbSet` persistence.

Stable rules must define:

- ordinary Active Record `query/find/save/delete` does not automatically
  attach or detach values in the tracker,
- tracked wrappers expose inherent `save/delete` methods so calls on
  `Tracked<T>` do not accidentally dereference to ordinary Active Record and
  leave stale tracker snapshots,
- `tracked.save(&db)` must synchronize the wrapper after immediate
  persistence so a later `save_changes()` does not update the same values
  again,
- `tracked.delete(&db)` must detach after successful immediate delete so a
  later `save_changes()` does not delete the same row again,
- mixing ordinary detached `entity.save(&db)` with a separately tracked row of
  the same identity remains outside automatic observation until the owned
  registry and identity-map work is completed,
- and duplicate identity conflicts keep using the documented tracker duplicate
  error.

There must not be two independent persistence routes for the same tracked row.

## Navigation Interaction

The first stable tracking cut does not need to persist relationship changes,
but it must define the boundary:

- `include(...)` and `include_many(...)` may materialize ordinary values
  without registering them automatically,
- explicit tracked loads may attach values to a root without marking the root
  as modified,
- relationship mutations inside `Navigation<T>` or `Collection<T>` are ignored
  unless a future graph-tracking task implements them,
- direct many-to-many remains outside stable tracking.

If identity map support is introduced for navigation loading, it must share the
same context-owned identity key as the tracker.

## Validation Required Before Stabilization

The stabilization work is not complete until the following pass:

- unit tests for state transitions, duplicate tracking, detach, clear, no-op
  saves and cancellation of `Added`,
- tests for deterministic save ordering,
- tests for all stable error cases,
- public `trybuild` fixtures from `sql_orm::prelude`,
- optional SQL Server runtime tests for insert, update, delete, soft delete,
  tenant, audit, rowversion and transactions,
- `cargo fmt --all --check`,
- `cargo check --workspace`,
- `cargo test --workspace`,
- `cargo clippy --workspace --all-targets --all-features`,
- `RUSTDOCFLAGS='-D warnings' cargo doc -p sql-orm --no-deps --all-features`.

When real SQL Server is not available, optional runtime tests may skip, but the
worklog must record that the stable claim still needs a real connection string
before release.

## Out Of Scope For The First Stable Cut

The following may remain out of scope if documented and tested as limits:

- direct many-to-many mutation persistence,
- automatic lazy loading,
- graph-wide cascade persistence from navigation collections,
- complete composite primary key support,
- transactions from pooled contexts until Etapa 22 pins one physical
  connection.

These limits must not be described as experimental once the stable contract is
set. They should be documented as supported exclusions.
